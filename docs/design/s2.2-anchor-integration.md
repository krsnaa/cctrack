## 0. Binding constraints (EM ruling msg 20565)

These are the load-bearing changes from v0.1 of this design note. Implementation
must satisfy each.

1. **Package layout**: `internal/usagescheduler` (new). Scheduler does NOT
   import `api` or `hub`. `cmd/serve` wires credentials, provider, store,
   scheduler lifecycle, and any broadcast callback.
2. **Snapshot extension belongs in S2.2**: add `FiveHourResetsAt` and
   `SevenDayResetsAt` `time.Time` fields to `usageprovider.Snapshot`; update
   `SCHEMA.md` to list the two new fields under the existing two. Allowlist
   stays at exactly four paths.
3. **`resets_at` parse format**: RFC3339 / RFC3339Nano ONLY. Other formats
   fail closed with `ErrSchemaDrift`; do NOT widen the probe to print value
   formats; do NOT paste raw values into chat/source/tests if a different
   format is observed — escalate to EM with no values.
4. **Shared observed-cost helper**: factor the cost-math out of the existing
   manual-sync POST `/window-anchors` handler into a helper. Scheduler and
   manual-sync handler must call the SAME helper. Cost window is
   `[resets_at - duration, observed_at)`, NOT `[now - duration, now)`.
5. **Anchor write fields**:
   - `SyncedAt` set explicitly from the snapshot's `Observed` clock
     (not auto-generated from server-side `time.Now()`).
   - `TimeLeftMinutes` derived from `resets_at - observed_at`.
   - Sub-minute future resets must NOT collapse to a permanently-stale
     zero-minute anchor. Pin in tests.
6. **Stale-snapshot rejection**: if either reset time is NOT after the
   observation time, treat the snapshot as unusable and do not write that
   anchor. Provider/schema-stale, not a valid zero-time-left sync.
7. **Trigger model**: APPROVED. Fetch on serve start, then at earliest
   observed reset + grace delay. No periodic polling. Backoff capped on
   provider/rate-limit failures. Auth/schema errors stop auto-refresh
   until restart or future explicit retry hook.
8. **Context lifecycle**: owned by `cmd/serve`. Shutdown cancels in-flight
   fetches and stops timers. Anchor writes are row-level; tests cover
   "at most one successful row before an error" classes.
9. **Scope freeze**: backend/internal only. NO new frontend UX or
   product-shape change. Manual percentage-entry stays available as
   fallback. Force-refresh-button idea is PM/PO scope and DEFERRED
   (was open question 4 in v0.1).
10. **LAD update after proof**: D4 changes to "read-only Claude Code
    credential sources" (was "file-only").

## 1. Goal

Replace the manual sync-percentage input flow with periodic automatic refresh
of `window_anchors` rows from the live `/api/oauth/usage` endpoint via the
already-proven `internal/usageprovider` adapter. After this slice, the dashboard
shows live 5h/7d window state without requiring the user to copy-paste values
from `claude.ai/usage`.

Manual sync UX remains as fallback (codex msg 20436 binding) when:
- token expired / credentials missing,
- provider unavailable / schema drift,
- non-Darwin host without keychain access AND no legacy file.

## 4. Snapshot extension

`usageprovider.Snapshot` gains two fields:

```go
type Snapshot struct {
    FiveHourUtilizationPercent int       // five_hour.utilization
    SevenDayUtilizationPercent int       // seven_day.utilization
    FiveHourResetsAt           time.Time // five_hour.resets_at (parsed)
    SevenDayResetsAt           time.Time // seven_day.resets_at (parsed)
    Observed                   time.Time
}
```

Decode: parse `resets_at` strings as RFC3339 (or whatever format the live
endpoint uses — schema discovery confirmed `string` type but exact format
needs a quick verify; if not RFC3339, the parser uses the actual format).
**Fail closed** on parse error: return `ErrSchemaDrift` rather than zero-time.

## 5. Anchor write mapping (per binding constraint #5/#6)

For each `WindowType` ("5h", "7d"), the scheduler builds a `WindowAnchor`
from the snapshot using the snapshot's own `Observed` clock as the
authoritative observation moment. Reset times that are not strictly after
`observed_at` mean the snapshot is unusable for that window.

```go
// pseudocode (shape only; see binding constraints 5 + 6)
func anchorFromSnapshot(snap Snapshot, win string, observedCost float64) (store.WindowAnchor, bool) {
    var resetsAt time.Time
    var pct float64
    switch win {
    case "5h":
        resetsAt = snap.FiveHourResetsAt
        pct = float64(snap.FiveHourUtilizationPercent)
    case "7d":
        resetsAt = snap.SevenDayResetsAt
        pct = float64(snap.SevenDayUtilizationPercent)
    }
    if !resetsAt.After(snap.Observed) {
        // Constraint #6: stale-snapshot rejection. Do not write a
        // permanently-stale zero-minute anchor; signal "skip this window."
        return store.WindowAnchor{}, false
    }
    // Constraint #5: derive TimeLeftMinutes from resets_at - observed_at,
    // not now-anything. Sub-minute future resets must round up rather than
    // collapse to zero (e.g. via Round vs Truncate, pin in tests).
    timeLeft := int(resetsAt.Sub(snap.Observed).Round(time.Minute).Minutes())
    if timeLeft < 1 && resetsAt.After(snap.Observed) {
        timeLeft = 1  // sub-minute future reset → 1, not 0
    }
    return store.WindowAnchor{
        SyncedAt:        snap.Observed.UTC().Format(time.RFC3339Nano), // explicit
        WindowType:      win,
        TimeLeftMinutes: timeLeft,
        AnthropicPct:    &pct,
        ObservedCost:    observedCost,
        // InferredCap left zero; SaveWindowAnchor computes it from pct + observedCost.
    }, true
}
```

### Shared observed-cost helper (binding constraint #4)

The cost-math currently lives inline in the manual-sync POST
`/api/v1/window-anchors` handler. Before S2.2 lands, factor it into a
shared helper:

```go
// internal/store (or a new internal/anchorcost), exact home is design choice
func ObservedCostForWindow(s *Store, windowDuration time.Duration, observedAt time.Time, resetsAt time.Time) (float64, error)
```

The cost window is `[resets_at - windowDuration, observed_at)` — NOT
`[now - windowDuration, now)`. Both the scheduler and the existing
manual-sync handler call this helper. Mirror points are documented with
the cross-reference contract: changes to either caller require a review
of the helper.

## 6. Scheduler shape

```go
type Scheduler struct {
    provider  Provider                 // usageprovider.Client in prod, fake in tests
    creds     CredentialSource         // credentials.Load, fakeable
    store     AnchorStore              // *store.Store, fakeable
    cost      ObservedCostFn           // computes ObservedCost from request log
    clock     func() time.Time         // for testability
    backoff   backoff.Strategy         // exponential, capped
}

func (s *Scheduler) Run(ctx context.Context) error
```

### Refresh trigger rules
1. **On serve start**: one fetch attempt; failure → backoff timer + log; success → write anchors and schedule next per reset times.
2. **At observed `resets_at` plus a small grace delay** (e.g. 30s) for whichever window resets next; both windows usually refresh together but the scheduler picks the earlier one.
3. **No periodic polling beyond reset-timing** — avoid hammering the endpoint. The exception is backoff after failure: try again in N seconds (exponential, capped at ~5 min).

### Single-flight
`usageprovider.Client` already serializes via mutex. The scheduler additionally
ensures no overlapping `Fetch` per process — a refresh in flight cancels-or-suppresses
a second trigger fired by reset-time arrival. (Mutex in scheduler + mutex in
provider is belt-and-suspenders; one is removable if EM prefers.)

### Context cancellation
The scheduler's `Run(ctx)` blocks until `ctx.Done()`. On cancellation:
- in-flight HTTP request is aborted (`http.Client` honors context)
- pending timer is stopped
- any partial state is discarded (no half-written anchor; `SaveWindowAnchor` is atomic per row)

### Backoff
On `ErrCredentialsMissing` / `ErrTokenExpired`: STOP the periodic loop and
surface a state to the UI ("provider unavailable, fix credentials"). Resume
on next `serve` start. Don't burn cycles polling an endpoint we know we
can't authenticate to.

On `ErrProviderUnavailable` / `ErrRateLimited`: exponential backoff (e.g.
30s → 60s → 2min → 5min cap). After 3 consecutive failures: surface state.

On `ErrSchemaDrift`: STOP the periodic loop, surface state, leave manual
fallback alive. Schema drift means the contract changed; we shouldn't keep
hitting the endpoint speculatively.

On `ErrUnauthorized` (401/403): STOP, surface state. Token may need refresh
which is Claude Code's job.

## 7. Error → state mapping (UX-relevant)

| Provider error                | UI state                  | Manual-sync available |
| ----------------------------- | ------------------------- | --------------------- |
| no error                      | `auto_fresh`              | yes (still works)     |
| stale anchor (window expired) | `auto_stale`              | yes                   |
| `ErrTokenExpired`             | `token_expired`           | yes                   |
| `ErrCredentialsMissing`       | `manual_anchor` (no auto) | yes                   |
| `ErrProviderUnavailable`      | `provider_unavailable`    | yes                   |
| `ErrSchemaDrift`              | `provider_unavailable`    | yes                   |
| `ErrUnauthorized`             | `token_expired`           | yes                   |
| no anchor at all              | `fallback_cascade`        | yes                   |

These are the six visible states from the F2 evidence requirements (item #637).

## 8. Test approach

All unit tests in `internal/usagescheduler` use fakes. No real HTTP, no real
DB, no real clock.

Test matrix:
1. Refresh-on-start writes both anchors when provider returns healthy snapshot.
2. Reset-time triggers next refresh at observed `resets_at + grace`.
3. Fetch failure → backoff schedule (verify timer intervals).
4. Context cancellation aborts in-flight refresh and stops timer.
5. Schema drift → loop stopped, no anchors written, state surfaced.
6. Token expired → loop stopped, no anchors written, state surfaced.
7. Single-flight: two concurrent triggers result in one fetch.
8. ObservedCost computation reuses manual-sync helper (verify by mocking the
   helper and asserting it's called with current rolling-window bounds).
9. Anchor write goes through `SaveWindowAnchor`; verify `InferredCap` is
   populated correctly (via the existing `pct/100` computation).

Property-style: time-zone handling for `resets_at` parse — verify RFC3339Nano
or whatever format is observed normalizes to UTC `time.Time`.

## 9. Open questions — resolution status

1. **`resets_at` exact format** — RESOLVED by EM msg 20565. Parse as
   RFC3339/RFC3339Nano only; fail closed on other formats. Probe NOT
   widened to print value formats. If a different format is observed in
   live smoke, escalate to EM with no values pasted.

2. **Backoff state location** — implicitly resolved by EM msg 20565 ("stop
   auto refresh until restart or a future explicit retry hook"). Reset on
   restart; no DB-persisted backoff state in v1.

3. **Window type names** — minor; keep mechanical mapping (`"5h"`/`"7d"` in
   the store layer; `five_hour`/`seven_day` at the wire layer). A single
   constant table in the scheduler maps between them.

4. **Manual sync coexistence as force-refresh trigger** — DEFERRED per EM
   msg 20565 ("PM/PO scope"). Manual percentage-entry stays available as
   the existing UX in v1. The "manual button becomes force-refresh trigger"
   behavior change is product-shape and waits for PO return.
